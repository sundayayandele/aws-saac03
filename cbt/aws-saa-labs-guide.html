IAM Identity Management

IAM Principle of least privilege · Users, Groups & Roles
▼
Create IAM users, groups, and roles applying the principle of least privilege to control access to AWS resources.

Steps

  1. Sign in to the AWS Management Console and navigate to IAM.
  2. Click User groups → Create group. Name it Developers and attach AmazonEC2ReadOnlyAccess.
  3. Click Users → Add users. Username: dev-user-01, enable console access with a custom password.
  4. On the Permissions step, add the user to the Developers group.
  5. Click Roles → Create role. Trusted entity: AWS service, use case: EC2.
  6. Attach AmazonS3ReadOnlyAccess, name the role EC2-S3-ReadOnly-Role, create it.
  7. Navigate to Policies → Create policy. Use the JSON editor to write a custom policy allowing only s3:GetObject on a specific bucket ARN.
  8. Log in as dev-user-01 and confirm EC2 read access works but write/terminate is denied.

Key Takeaways

  • Groups simplify permission management across many users.
  • Roles avoid embedding long-term credentials in applications.
  • Always grant only the minimum permissions required (least privilege).

VPC Network Isolation

VPCEC2Public & private subnets · NAT Gateway · Security Groups
▼
Design a custom VPC with public and private subnets, NAT Gateway, and Security Groups for a two-tier application.

Steps

  1. Go to VPC → Create VPC. Name: LabVPC, CIDR: 10.0.0.0/16.
  2. Create two subnets in LabVPC: Public-Subnet-1A (CIDR 10.0.1.0/24) and Private-Subnet-1A (CIDR 10.0.2.0/24), both in us-east-1a.
  3. Enable Auto-assign public IPv4 on Public-Subnet-1A.
  4. Create an Internet Gateway (LabIGW) and attach it to LabVPC.
  5. Create route table Public-RT: add route 0.0.0.0/0 → LabIGW. Associate with Public-Subnet-1A.
  6. Allocate an Elastic IP. Create a NAT Gateway in Public-Subnet-1A using the Elastic IP.
  7. Create route table Private-RT: add route 0.0.0.0/0 → NAT Gateway. Associate with Private-Subnet-1A.
  8. Create Security Group WebSG: allow HTTP (80) from 0.0.0.0/0, SSH (22) from your IP only.
  9. Launch an EC2 instance in Public-Subnet-1A with WebSG. SSH in and confirm internet access.
  10. Launch a second EC2 in Private-Subnet-1A. From the public instance, SSH to the private instance, then run curl https://aws.amazon.com — traffic routes through NAT.

Key Takeaways

  • Security Groups are stateful — return traffic is automatically allowed.
  • NACLs are stateless — you must explicitly allow inbound and outbound rules.
  • NAT Gateway lets private instances reach the internet without being publicly reachable.

S3 Data Encryption & Security

S3KMSSSE-KMS · Bucket Policies · Enforcement
▼
Implement SSE-KMS server-side encryption and enforce it via Bucket Policies for access control.

Steps

  1. Navigate to KMS → Customer managed keys → Create key. Choose Symmetric, Encrypt and decrypt. Name: S3LabKey.
  2. Add yourself as key administrator and user. Finish creating the key.
  3. Go to S3 → Create bucket. Name: lab-encrypted-bucket-<unique-id>, Block Public Access on.
  4. Open bucket → Properties → Default encryption → Edit. Select SSE-KMS, choose S3LabKey. Save.
  5. Upload a test file — check object properties show aws:kms encryption.
  6. Go to Permissions → Bucket policy. Paste a policy that denies s3:PutObject when the request header s3:x-amz-server-side-encryption is not aws:kms.
  7. Attempt to upload via AWS CLI without specifying encryption — confirm it is denied.
  8. Upload with --sse aws:kms — confirm success.

Key Takeaways

  • SSE-KMS provides a full audit trail via CloudTrail for every key usage event.
  • Bucket Policies can enforce encryption-at-rest on all incoming objects.
  • KMS Customer Managed Keys give you full control over key rotation and access.

Web Security with WAF & Shield

WAFShieldALBDDoS protection · Web ACL · Managed Rules
▼
Protect an Application Load Balancer from common web exploits (SQLi, XSS) and DDoS attacks.

Steps

  1. Deploy an EC2 instance with Apache/Nginx in a public subnet. Confirm it serves a web page.
  2. Create an Application Load Balancer (internet-facing) across two AZs. Register the EC2 as a target. Confirm the ALB DNS serves your page.
  3. Go to WAF & Shield → Web ACLs → Create web ACL. Resource type: Regional. Associate with your ALB.
  4. Add managed rule group: AWSManagedRulesCommonRuleSet (blocks SQLi, XSS, etc.).
  5. Add managed rule group: AWSManagedRulesAmazonIpReputationList (blocks malicious IPs).
  6. Set default action to Allow. Create the Web ACL.
  7. Test WAF: send a simulated SQLi string in a query parameter (e.g., ?id=1' OR '1'='1). Confirm WAF returns HTTP 403.
  8. Go to Shield — confirm Shield Standard is active (automatic for all AWS customers).
  9. Review WAF sampled requests in the console to see blocked traffic details.

Key Takeaways

  • WAF inspects HTTP/HTTPS at Layer 7 — it can block based on request content.
  • Shield Standard is free and automatic; Shield Advanced adds DDoS response team support.
  • Managed rule groups save time — AWS maintains them against new threat signatures.

Cross-Account Access with IAM Roles

IAMSTSTrust policies · AssumeRole · Temporary credentials
▼
Configure an IAM role to allow secure, temporary access to resources in a different AWS account without sharing credentials.

Steps

  1. In Account A, go to IAM → Roles → Create role. Trusted entity: AWS account. Enter Account B's Account ID.
  2. Attach AmazonS3ReadOnlyAccess. Name the role CrossAccountS3Role. Copy the Role ARN.
  3. In Account B, create a custom IAM policy allowing sts:AssumeRole on the Role ARN from Account A. Attach it to a user.
  4. Log in as the Account B user and run via AWS CLI:
aws sts assume-role \
  --role-arn "arn:aws:iam::<AccountA-ID>:role/CrossAccountS3Role" \
  --role-session-name "CrossAccountSession"
  1. Export the returned temporary credentials as environment variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN).
  2. Run aws s3 ls — confirm you can list Account A's S3 buckets.
  3. Try a write operation — confirm it is denied (read-only policy enforced).

Key Takeaways

  • Cross-account roles use STS to issue temporary credentials — no static keys shared between accounts.
  • The trust policy controls who can assume the role; the permission policy controls what they can do.
  • This is the AWS best practice for cross-account access in organizations.

Multi-AZ Web Application

EC2ALBRoute 53High availability · Load balancing
▼
Deploy a web server across two Availability Zones using an ALB for high availability and zero-downtime failover.

Steps

  1. Launch two EC2 instances: one in us-east-1a, one in us-east-1b. Install Apache on each; configure each to display its AZ name in the default HTML page.
  2. Create an Application Load Balancer (internet-facing) spanning both AZs. Create a target group WebTG and register both instances.
  3. Set the ALB listener (HTTP:80) to forward to WebTG.
  4. Access the ALB DNS name in a browser. Refresh multiple times — observe traffic alternating between instances.
  5. Stop one instance. Refresh — the ALB routes all traffic to the healthy instance within ~30 seconds.
  6. In Route 53, create an A record (Alias) pointing to the ALB DNS name in your hosted zone.
  7. Access via your domain — confirm traffic is served and resilient to single-AZ failure.

Key Takeaways

  • ALB health checks automatically remove unhealthy instances from rotation.
  • Route 53 Alias records resolve directly to ALB with no extra DNS hop.
  • Spreading across AZs protects against data center failures.

Auto Scaling for Demand

ASGCloudWatchTarget tracking · Scale-out/in · CPU stress test
▼
Configure an Auto Scaling Group with target tracking to automatically add or remove EC2 instances based on CPU utilization.

Steps

  1. Create a Launch Template: AMI = Amazon Linux 2, type t3.micro. User data installs the stress tool.
  2. Go to Auto Scaling Groups → Create Auto Scaling group. Select your launch template, VPC, and public subnets in both AZs.
  3. Attach to the ALB target group from Lab 6.
  4. Set capacity: Desired: 2, Minimum: 1, Maximum: 4.
  5. Add a Target Tracking Scaling Policy: metric = Average CPU Utilization, target = 50%.
  6. SSH into an instance and run a CPU stress test:
stress --cpu 4 --timeout 300
  1. Monitor CloudWatch ASG metrics — within ~3 minutes, observe new instances launching (scale-out).
  2. Stop the stress test. After ~10 minutes observe scale-in back to minimum.

Key Takeaways

  • Target tracking automatically adjusts capacity — no manual thresholds needed.
  • Cooldown periods prevent thrashing during rapid scaling events.
  • ALB automatically registers and deregisters ASG instances as they scale.

RDS Multi-AZ & Failover

RDSAuroraSynchronous replication · Automatic failover · Zero data loss
▼
Launch an RDS MySQL instance with Multi-AZ enabled and simulate a primary node failure to observe automatic failover.

Steps

  1. Go to RDS → Create database. Engine: MySQL. Enable Multi-AZ deployment.
  2. DB identifier: lab-mysql-db. VPC: LabVPC. Subnet group covering two AZs. Security group: allow MySQL (3306) from your EC2.
  3. Note the endpoint DNS and which AZ is primary after creation.
  4. Connect from EC2: mysql -h <endpoint> -u admin -p. Create a test table, insert rows.
  5. In the RDS console → select DB → Actions → Reboot → check Reboot With Failover. Confirm.
  6. Monitor the MySQL connection — it drops briefly, then reconnects automatically (same endpoint DNS).
  7. After reboot completes, check the console — the primary AZ has changed.

Key Takeaways

  • Multi-AZ = synchronous replication → zero data loss on failover (HA).
  • The standby is not readable. Use Read Replicas for read scaling (asynchronous).
  • Application code using the RDS DNS endpoint requires no changes during failover.

S3 Cross-Region Replication

S3CRR · Versioning · Disaster recovery
▼
Automate replication of S3 objects to a different AWS region for disaster recovery purposes.

Steps

  1. Create source bucket lab-source-crr-<id> in us-east-1. Enable Versioning (required for CRR).
  2. Create destination bucket lab-dest-crr-<id> in us-west-2. Enable Versioning.
  3. In the source bucket → Management → Replication rules → Create replication rule. Name: CRR-to-West.
  4. Source: apply to all objects. Destination: select lab-dest-crr-<id>. Let AWS create a new IAM role. Save.
  5. Upload a test file to the source bucket.
  6. Wait 30–60 seconds, then check the destination bucket in us-west-2 — the object should appear.
  7. Check the object's Replication status — should show COMPLETED.
  8. Enable Delete marker replication and test: delete an object in the source — observe the delete marker in the destination.

Key Takeaways

  • CRR requires versioning enabled on both source and destination buckets.
  • Supports cross-account replication for additional isolation.
  • Replication is asynchronous — objects appear in the destination within seconds to minutes.

Route 53 Failover Routing

Route 53Health checks · Failover policy · DNS routing
▼
Implement Route 53 health checks and failover routing to redirect DNS traffic automatically during an outage.

Steps

  1. Set up two web servers: Primary in us-east-1, Secondary in us-west-2 (or S3 static websites).
  2. Go to Route 53 → Health Checks → Create health check. Protocol: HTTP, IP = primary server, path: /.
  3. Create DNS A record (Primary): routing policy = Failover, type = Primary, associate the health check.
  4. Create DNS A record (Secondary): routing policy = Failover, type = Secondary (no health check needed).
  5. Access your domain — traffic should reach the primary server.
  6. Stop the primary web server (or block port 80). Wait ~30–60 seconds for the health check to go Unhealthy.
  7. Re-query the DNS — Route 53 now returns the secondary server's IP.
  8. Restart the primary — the health check recovers and Route 53 fails back automatically.

Key Takeaways

  • Failover routing requires a health check attached to the primary record.
  • Route 53 TTL affects how quickly DNS changes propagate to clients after failover.
  • Health check interval default is 30 seconds — adjust for faster detection.

Serverless API with Lambda

LambdaAPI GatewayServerless · Auto-scaling · Pay-per-use
▼
Build a RESTful API using API Gateway and Lambda to handle variable traffic without managing servers.

Steps

  1. Go to Lambda → Create function → Author from scratch. Name: HelloWorldFunction, runtime: Python 3.12.
  2. Paste this handler code and click Deploy:
import json
def lambda_handler(event, context):
    name = event.get('queryStringParameters', {}).get('name', 'World')
    return {
        'statusCode': 200,
        'body': json.dumps({'message': f'Hello, {name}!'})
    }
  1. Go to API Gateway → Create API → HTTP API. Add integration: Lambda → HelloWorldFunction. Route: GET /hello. Stage: $default. Create.
  2. Copy the Invoke URL. Open a browser: <invoke-url>/hello?name=AWS — verify the JSON response.
  3. Go to Lambda → Monitor tab → view invocation metrics in CloudWatch Logs.
  4. Run a concurrent load test to see Lambda scale automatically with no server management.

Key Takeaways

  • Lambda charges only for compute time (millisecond precision) — zero cost at idle.
  • API Gateway handles auth, throttling, rate limiting, and routing.
  • Serverless removes the need to manage, patch, or right-size servers.

DynamoDB Global Tables

DynamoDBMulti-region replication · Active-active · Low latency
▼
Create a NoSQL database with active-active multi-region replication for low-latency global access.

Steps

  1. Go to DynamoDB → Create table. Name: GlobalUsersTable, partition key: userId (String).
  2. Wait for status Active. Go to the Global Tables tab → Create replica → add region us-west-2.
  3. In us-east-1, insert item: userId = user-001, name = Alice.
  4. Switch to us-west-2 console → open the replica → confirm user-001 replicated (typically milliseconds).
  5. Update the item in us-west-2: change name = Alice Updated.
  6. Switch back to us-east-1 — confirm the update is reflected (last-writer-wins).
  7. Measure latency: query the table from an EC2 in each region and compare response times.

Key Takeaways

  • Global Tables use active-active replication — writes accepted in any region.
  • Conflict resolution is last-writer-wins based on timestamp.
  • Ideal for globally distributed apps requiring single-digit millisecond reads.

CloudFront Content Delivery

CloudFrontS3CDN · Edge caching · OAC · Cache hit/miss
▼
Configure a CloudFront CDN distribution to cache static content at edge locations and reduce latency globally.

Steps

  1. Create S3 bucket lab-cloudfront-origin-<id> in us-east-1. Upload static files. Keep bucket private.
  2. Go to CloudFront → Create distribution. Origin: your S3 bucket. For Origin access, select Origin Access Control (OAC) → create new OAC.
  3. Copy the generated S3 bucket policy shown by CloudFront and apply it to your bucket's permissions.
  4. Cache behavior: policy = CachingOptimized, viewer protocol = Redirect HTTP to HTTPS. Create.
  5. Wait ~5–10 minutes for status: Enabled. Access: <distro-id>.cloudfront.net/index.html.
  6. First request: check response header X-Cache = Miss from cloudfront (served from S3).
  7. Second request to same URL: X-Cache = Hit from cloudfront (served from edge cache).
  8. Enable CloudFront access logs to an S3 bucket. Review cache hit ratios.

Key Takeaways

  • OAC is the modern recommended way to restrict S3 access to CloudFront only.
  • Cache hits reduce latency dramatically and reduce S3 request costs.
  • CloudFront has 400+ edge locations — content is served from the nearest PoP.

ElastiCache for Performance

ElastiCacheRedis · Cache-aside pattern · TTL · DB offload
▼
Implement a Redis cluster to cache frequent database query results and dramatically reduce database load.

Steps

  1. Go to ElastiCache → Redis clusters → Create Redis cluster. Easy create, node type cache.t3.micro. Name: LabRedisCluster. VPC: LabVPC.
  2. Security group: allow Redis port 6379 from your EC2 security group. Wait for status Available.
  3. Install Redis CLI on your EC2:
sudo amazon-linux-extras install redis6
  1. Connect: redis-cli -h <primary-endpoint> -p 6379
  2. Set a key with TTL: SET db:user:1 '{"name":"Alice"}' EX 300
  3. Get the key: GET db:user:1 — instant retrieval from cache.
  4. Implement cache-aside in Python: check Redis first; on miss, query DB, then write to Redis. Measure latency difference.
  5. Wait for TTL (300s) to expire — GET returns nil, forcing a DB re-query (cache miss).

Key Takeaways

  • Cache-aside (lazy loading) only caches what is actually read — efficient memory use.
  • TTL prevents stale data from persisting indefinitely in the cache.
  • ElastiCache can reduce DB load by 80%+ for read-heavy workloads.

VPC Peering & Transit Gateway

VPCTransit GatewayPrivate connectivity · Hub-and-spoke · Transitive routing
▼
Connect multiple VPCs to enable private communication — first via VPC Peering, then via Transit Gateway for transitive routing.

Part A — VPC Peering

  1. Create VPC-A (10.1.0.0/16) and VPC-B (10.2.0.0/16), each with a subnet and an EC2 instance.
  2. Go to VPC Peering Connections → Create peering connection. Requester: VPC-A, Accepter: VPC-B. Accept the request.
  3. Update route tables: VPC-A → add 10.2.0.0/16 via peering; VPC-B → add 10.1.0.0/16 via peering.
  4. Update Security Groups to allow ICMP/SSH from the other VPC's CIDR. Ping between EC2s.

Part B — Transit Gateway

  1. Create VPC-C (10.3.0.0/16) with an EC2 instance.
  2. Go to Transit Gateways → Create Transit Gateway. Note the TGW ID.
  3. Create Transit Gateway Attachments for VPC-A, VPC-B, and VPC-C.
  4. Add routes in each VPC's route table pointing to the other two VPCs' CIDRs via the TGW.
  5. Ping between all three EC2 instances using private IPs — all should succeed.

Key Takeaways

  • VPC Peering is non-transitive — A↔B and B↔C does not give A↔C connectivity.
  • Transit Gateway enables transitive routing — a central hub for thousands of VPCs.
  • Use TGW for complex multi-VPC architectures; VPC Peering for simple point-to-point connections.

S3 Lifecycle Policies

S3 LifecycleStorage tiers · Glacier · Automated cost reduction
▼
Automatically transition aging data through S3 storage classes to minimize storage costs over time.

Steps

  1. Create bucket lab-lifecycle-<id>. Enable versioning. Upload several test objects.
  2. Go to Management → Lifecycle rules → Create lifecycle rule. Name: DataTiering. Apply to all objects.
  3. Add transitions: 30 days → S3 Standard-IA; 90 days → S3 Intelligent-Tiering; 180 days → S3 Glacier Flexible Retrieval; 365 days → S3 Glacier Deep Archive.
  4. Add expiration: expire current object versions after 730 days.
  5. Add noncurrent version expiration: delete noncurrent versions after 90 days.
  6. Save the rule and review the rule summary in the console.
  7. Manually change an existing object's storage class to S3 Standard-IA and compare per-GB pricing.

Key Takeaways

  • Standard-IA minimum storage duration: 30 days. Glacier Deep Archive: 180 days.
  • Lifecycle rules run automatically — set it once, save forever.
  • Intelligent-Tiering is ideal for unpredictable access patterns — AWS optimizes tiers automatically.

Spot Instance Fleet

EC2 SpotUp to 90% cost savings · Mixed Instances Policy · Fault-tolerant workloads
▼
Use EC2 Spot Instances for fault-tolerant workloads to achieve up to 90% savings compared to On-Demand pricing.

Steps

  1. Go to EC2 → Spot Requests → Request Spot Instances. AMI: Amazon Linux 2, type: c5.large. Max price: default (on-demand cap).
  2. Fleet settings: Maintain target capacity = 2 instances. Interruption behavior: Terminate. Note estimated savings vs On-Demand.
  3. Launch the fleet. Observe two Spot instances running in EC2 → Instances.
  4. Create an Auto Scaling Group with a Mixed Instances Policy:
On-Demand base: 1 instance (always reliable)
Spot percentage: 80% of additional capacity
  1. Confirm the ASG launches a mix of On-Demand and Spot instances.
  2. Review the Spot Instance Advisor to select instance types with low interruption frequency.
  3. Add a Spot interruption handler in user data to gracefully drain tasks on 2-minute warning.

Key Takeaways

  • Spot Instances can be interrupted with a 2-minute warning — design workloads to handle this gracefully.
  • Best for stateless, batch, CI/CD, big data, and rendering workloads.
  • Mixed Instances Policy = savings + reliability baseline from On-Demand base capacity.

EFS vs. EBS for Shared Storage

EBSEFSBlock vs. file storage · Shared access · Cost comparison
▼
Compare EBS block storage and EFS elastic file system costs, use cases, and performance characteristics.

Part A — EBS (Block Storage)

  1. Launch an EC2 instance. Attach a second EBS volume (gp3, 10 GB) in the same AZ.
  2. SSH in. Format and mount:
sudo mkfs -t ext4 /dev/xvdf
sudo mkdir /data && sudo mount /dev/xvdf /data
  1. Write a large test file: dd if=/dev/zero of=/data/testfile bs=1M count=500. Note EBS is tied to one AZ and one instance.

Part B — EFS (Elastic File System)

  1. Go to EFS → Create file system, select LabVPC. Create mount targets in each subnet/AZ. Security group: allow NFS (2049) from EC2.
  2. On two EC2 instances (different AZs), install and mount EFS:
sudo yum install -y amazon-efs-utils
sudo mkdir /efs && sudo mount -t efs <fs-id>:/ /efs
  1. On Instance 1: echo "hello from AZ-1" > /efs/shared.txt
  2. On Instance 2: cat /efs/shared.txt — confirms shared, concurrent access across AZs.

Key Takeaways

  • EBS: ~$0.08/GB/mo (gp3). Single AZ. Single instance. Low latency. Use for DBs and boot volumes.
  • EFS: ~$0.30/GB/mo (Standard). Multi-AZ. Shared. Auto-scales. Use for shared application data.
  • EFS is more expensive per GB but eliminates the need to manage multiple volumes.

Aurora Serverless v2

Aurora ServerlessACU auto-scaling · Pay-per-use · Variable workloads
▼
Deploy an Aurora Serverless v2 database that scales capacity automatically on demand to minimize costs for variable workloads.

Steps

  1. Go to RDS → Create database. Engine: Aurora (MySQL Compatible). Edition: Aurora Serverless v2.
  2. DB cluster identifier: lab-aurora-serverless. Instance configuration: Serverless v2.
  3. Set Minimum ACUs: 0.5, Maximum ACUs: 16 (1 ACU ≈ 2 GB RAM). VPC: LabVPC. Create.
  4. Connect from EC2: mysql -h <cluster-endpoint> -u admin -p. Create a table and insert rows.
  5. With load idle, open CloudWatch → Aurora metric ServerlessDatabaseCapacity — observe it near minimum ACU.
  6. Run a load test inserting 10,000 rows in a loop (Python mysql-connector or sysbench).
  7. Watch ServerlessDatabaseCapacity — observe ACU scaling up within seconds to handle the load.
  8. Stop the load test — observe scale-down back toward minimum ACU.

Key Takeaways

  • Aurora Serverless v2 scales in 0.5 ACU increments within seconds — fine-grained and fast.
  • Ideal for variable, unpredictable, dev/test, and intermittent production workloads.
  • You only pay for ACU-hours consumed — not provisioned capacity sitting idle.
