🧪 AWS SAA-C03 Hands-On Labs
Step-by-step practical training for exam success

These labs are directly aligned with the SAA-C03 exam domains. Complete them to build real-world skills and reinforce architectural concepts.

🔒 Lab 1: Secure VPC with Public/Private Subnets & NAT Gateway Security · Resilience

Build a production-ready VPC with public and private subnets, enabling secure outbound internet access for private instances.

1 Create a VPC with CIDR 10.0.0.0/16.
2 Create two public subnets (10.0.1.0/24, 10.0.2.0/24) in different AZs.
3 Create two private subnets (10.0.3.0/24, 10.0.4.0/24).
4 Create an Internet Gateway and attach to VPC.
5 Configure route tables: public subnets → IGW; private subnets → NAT Gateway.
6 Launch an EC2 instance in a private subnet and test outbound connectivity.
# After NAT Gateway is created, SSH into a public bastion, then to private instance:
ssh -i key.pem ec2-user@<bastion-ip>
ssh -i key.pem ec2-user@<private-ip>
curl https://checkip.amazonaws.com   # Should return NAT Gateway EIP
🌍 Lab 2: Host a Static Website with S3 & CloudFront (OAC) High-Performance · Cost

Secure and accelerate static content delivery using S3 static website hosting and CloudFront with Origin Access Control.

1 Create an S3 bucket, enable static website hosting, upload index.html.
2 Create a CloudFront distribution with the S3 bucket as origin.
3 Use Origin Access Control (OAC) to restrict bucket access to CloudFront only.
4 Update bucket policy to allow CloudFront service principal.
5 Test website via CloudFront domain name.
💡 Exam insight: OAC replaced OAI and is now the recommended method for securing S3 origins.
📈 Lab 3: High-Availability Web App with ALB + Auto Scaling Resilience · Performance

Deploy a fault-tolerant application that automatically scales based on CPU load.

1 Create a launch template with Amazon Linux 2, user-data script to install httpd.
2 Create an Auto Scaling group spanning at least two AZs.
3 Create an Application Load Balancer (ALB) and attach to ASG.
4 Configure target tracking scaling policy (CPU utilization 50%).
5 Simulate load using `stress` tool and observe scaling events.
🗄️ Lab 4: Deploy a Highly Available RDS Database Resilience

Set up an Amazon RDS for MySQL with Multi-AZ deployment for automatic failover.

1 Create a DB subnet group covering at least two AZs.
2 Launch RDS MySQL instance with Multi-AZ enabled.
3 Connect to the RDS endpoint and create a test table.
4 (Optional) Initiate a reboot with failover to observe seamless switch.
⚠️ Ensure you delete the RDS instance after the lab to avoid ongoing costs.
⚡ Lab 5: Build a Serverless REST API Cost-Optimized · Performance

Create a simple serverless API using AWS Lambda and API Gateway, a common exam pattern.

1 Create a Lambda function (Node.js/Python) that returns a JSON response.
2 Create an API Gateway REST API with a GET method.
3 Integrate the GET method with the Lambda function.
4 Deploy the API to a stage and test the endpoint.
🔐 Lab 6: Private Access to S3 via VPC Endpoint Security

Keep S3 traffic within the AWS network, a critical security best practice.

1 Create a Gateway VPC Endpoint for S3 in your VPC.
2 Update route table of private subnets to point to the endpoint.
3 Launch an EC2 instance in a private subnet (no internet).
4 Use AWS CLI to list S3 buckets (`aws s3 ls`) – should succeed privately.
🔗 This lab demonstrates why private connectivity is preferred over NAT Gateway for S3 access.
📊 Lab 7: Monitoring & Alerting with CloudWatch Resilience · Performance

Set up proactive monitoring for your EC2 instances and receive email notifications.

1 Create an SNS topic and subscribe your email.
2 Create a CloudWatch alarm for EC2 CPU utilization > 80% for 5 minutes.
3 Configure alarm action to publish to SNS topic.
4 Generate load on EC2 and verify email notification.
👤 Lab 8: IAM Roles and Cross-Account Access Security

Grant permissions from one AWS account to resources in another, a key security pattern.

1 In Account A, create an IAM role with a trust policy allowing Account B.
2 Attach a policy granting S3 read access.
3 In Account B, assume the role using `aws sts assume-role`.
4 Use temporary credentials to list S3 buckets in Account A.
